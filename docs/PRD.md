## 一、产品定位与目标

### 1.1 一句话定位
一个"任务—工时—效能—绩效"闭环管理工具，解决部门内部 **人怎么管、活怎么分、钱怎么算** 的问题。

### 1.2 核心目标与可证伪指标

| 目标 | 解决的痛点 | 可证伪指标（失败阈值） |
|------|----------|----------------------|
| **量化管理** | 凭印象评工作量、忙闲不均 | **任务录入率 < 60%**（第 4 周末） |
| **行为转化** | 员工不愿意用 | **每日确认率 < 50%**（第 8 周末） |
| **公平分配** | 年终奖大锅饭 | **评分申诉率 > 20%** 或 **主观公平感提升 < 10%**（季度匿名调研） |
| **数据可信** | PM 敷衍录入 | **PM 预估工时方差 < 20%**（说明在填默认值） |

> 任何一个指标击穿失败阈值，对应阶段不进入下一阶段。

### 1.3 非目标（明确不做）

#### 范围反目标
- ❌ 不做对外合同收款、票据、回款
- ❌ 不做客户 CRM
- ❌ 不做财务工资结算（仅产出虚拟产值参考）
- ❌ 不做跨部门协作

#### 文化反目标（关键，写给员工看）
- ❌ **不是监工系统** —— 不记录员工电脑操作、不截屏、不计算"摸鱼时间"
- ❌ 不替代 IM —— 任务讨论仍在微信/钉钉
- ❌ 不替代设计软件 —— 不集成 CAD/Revit 操作记录
- ❌ 不替代请假流程 —— 仍走 OA

> 这一点必须在员工第一次使用时通过引导页明确告知，否则推行成本极高。

### 1.4 关键假设（任何一条不成立则项目高风险）

| 假设 | 风险等级 | 验证方式 |
|------|---------|---------|
| 员工愿意每日勾选确认任务 | 🔴 高 | Phase 1 末看每日确认率 |
| PM 愿意认真录预估工时 | 🔴 高 | Phase 1 末看预估工时方差 |
| 虚拟产值规则能服众 | 🔴 高 | Phase 3 试运行期申诉率 |
| 工时数据真实 | 🟡 中 | 与项目里程碑反推核对 |
| 部长长期推行 | 🟡 中 | 制度文件留底（管理层背书） |

## 二、用户角色与权限

> 复用现有 `permissions.ts` 三角色 RBAC。

| 角色 | 范围 | 关键权限 |
|------|------|---------|
| **部门管理员（ADMIN）** | 全部门 | 配置项目类型 / 任务分类 / 虚拟产值规则；导出全员数据；处理申诉 |
| **项目负责人（PROJECT_LEAD）** | 自己负责的项目 | 录入 / 编辑任务；查看项目内人员负荷；填写延期 / 插队原因 |
| **普通员工（MEMBER）** | 仅本人 | 查看 / 确认 / 完成任务；提交工时；查看个人效能；提交申诉 |

**字段级敏感数据**：身份证号、电话由权限模块脱敏。

## 三、分阶段功能（重切后的 MVP 路线）

> ⚠️ **核心原则：每一阶段都是独立可用的产品。前一阶段不达指标，不进入下一阶段。**

---

### 🚀 Phase 1（4 周）· 解决"看不见"

> **目标**：让派活和忙闲变得可见，不引入任何评分。

#### 1.1 任务管理（核心）
- **任务录入**（PM）：项目 → 任务详情 → 工时与时间，≤ 3 步
  - 必填：名称、所属项目（可选非项目）、专业、责任人、预估工时、计划开始 / 结束、优先级
  - 插队任务：勾选时强制填"插队原因"，系统自动顺延同责任人未开始任务
- **状态机**：未开始 / 进行中 / 已完成 / 逾期
  - **lazy 刷新**：用户访问时按 `plannedEnd < now()` 自动判定逾期（避免 cron 依赖）
  - PM 在员工勾完成后 24h 内确认
- **任务列表**：按项目 / 责任人 / 状态过滤；逾期飘红

#### 1.2 团队负荷热力图（PM/ADMIN）
- X 轴：未来 4 周（按周聚合）
- Y 轴：人员（按专业分组）
- 单元格颜色：绿 < 70% / 黄 70–90% / 红 > 90%
- 悬停看人员当周已派任务

#### 1.3 个人任务页（员工）
- 我的任务列表
- 一键勾"完成"
- 一键报"延期 + 原因"

#### Phase 1 验收门
- ✅ 试点专业（建筑）任务录入率 ≥ 60%
- ✅ 系统稳定性：P95 < 1.5s，无数据丢失
- ❌ **不达标 → 暂停推进，先解决推行问题**

---

### 🔔 Phase 2（4 周）· 解决"管不了"

> **前提**：Phase 1 验收通过。
> **目标**：让流程脱离口头沟通，异常自动浮上来。

#### 2.1 每日任务确认（员工）
- 移动端响应式（不做 PWA）
- 每日登录看"今日任务"，勾选确认（承诺今天会做）
- 仅暴露 2 个按钮：✅ 完成 / ⚠️ 报告延期

#### 2.2 异常提醒
| 触发 | 通知 | 要求 |
|------|------|------|
| 连续 3 天有任务未勾完成 | PM + 员工 | PM 填"处理措施" |
| 单任务延期 > 24h | PM + 员工 | 同上 |
| 个人本周饱和度 > 110% | PM + ADMIN | PM 评估调度 |
| 任务被多次插队（≥3 次） | ADMIN | ADMIN 介入 |

通道：站内消息（必须）+ 飞书机器人（复用 OpenClaw NAS 上现有 webhook）

#### 2.3 申诉流程
- 员工对任务状态 / 工时数据有异议 → 提交申诉
- ADMIN 3 个工作日内反馈
- 处理结果留痕

#### Phase 2 验收门
- ✅ 员工每日确认率 ≥ 50%
- ✅ 异常提醒响应率（PM 24h 内填处理措施）≥ 70%
- ❌ **不达标 → 不上 Phase 3。评分公式没有可信数据源**

---

### 📊 Phase 3（按需，至少 8 周后）· 解决"算不清"

> **硬前置**：Phase 1+2 数据真实可信，PM 预估工时方差 ≥ 20%（证明在认真填）。
> **目标**：把数据沉淀为绩效证据。

#### 3.1 虚拟产值规则（ADMIN 可配置）
| 任务类型 | 公式 | 默认参数 |
|---------|------|---------|
| 施工图 | 合同额 × 比例 | 1.0% |
| 方案（中标） | 合同额 × 比例 | 0.5% |
| 方案（未中标） | 建筑面积 × 单价 | 医院 2 元/㎡ |
| 驻场配合 | 实际天数 × 日薪 | 800 元/天 |
| 可研 / 报建 | 固定额度 | 由 ADMIN 配 |
| 其他 | 工时 × 时薪 | 100 元/h |

虚拟产值按 Task 责任人 × 占项目总工时比例分配。

#### 3.2 效能评分（0–100，权重可配置）
```
评分 = 任务完成率 × 30%
     + (1 − |工时偏差率|) × 20%
     + (1 − 逾期次数 / 月任务总数) × 20%
     + 虚拟产值排名分位数 × 30%
```

#### 3.3 试运行期（强制）
- **第 1 个完整月**：评分仅展示，不挂奖金
- **评审会**：员工 / PM / ADMIN 一起调权重
- **第 2 个月起**：才正式挂钩奖金系数（前 20% × 1.5、后 20% × 0.8）

#### 3.4 配套
- 申诉延伸到评分（独立通道）
- 审计日志（任何评分输入修改留痕）
- 月度 / 季度 Excel 导出

#### Phase 3 验收门
- ✅ 评分申诉率 ≤ 20%
- ✅ 季度匿名调研：主观公平感提升 ≥ 10%

## 四、运作规则

| 规则 | 说明 |
|------|------|
| **任务录入强制化** | 部门内一切任务必须走平台。口头派活作为例外，需 24h 内补录 |
| **工时缓冲建议** | 系统在 PM 录入预估工时时弹提示："建议 +10–15%"；不强制 |
| **异议处理 SLA** | 申诉 3 个工作日内必须反馈，超期升级 ADMIN |
| **数据不可删除** | Task / WorkLog 状态变更只能用"撤销"或"修订"，原记录留痕 |

## 五、非功能需求

| 项 | 要求 |
|----|------|
| **轻量化** | 任务录入 ≤ 3 步；移动端核心操作 ≤ 2 次点击 |
| **数据安全** | Zeabur 部署 + 部门 IP 白名单（v1.1）；密码 bcrypt；敏感字段角色级脱敏 |
| **移动端** | 试点期响应式即可；PWA 推迟到 Phase 2 末按需评估 |
| **性能** | 列表页 P95 < 800ms；看板 P95 < 1.5s |
| **可观测** | Phase 3 起，关键操作写审计日志 |
| **cron 替代** | 使用 lazy 扫描（用户访问时刷新自身任务状态），零外部依赖 |

## 七、关键决策（已拍板）

| 决策 | 结论 |
|------|------|
| cron 实现 | **lazy 扫描**（零外部依赖） |
| Milestone 处理 | **保留 + 并存**（不升级为 Task） |
| MVP 范围 | **Phase 1+2 优先**，Phase 3 按需 |
| 评分挂奖金时机 | **至少跑 1 个月观察期** |
| TaskStatus 类型 | **String + 应用层校验**（避免 enum 迁移成本） |
| 移动端 | **响应式优先**，PWA 按需 |

## 附录 A · 关键术语

- **预估工时**：PM 录入任务时给出的估算
- **实际工时**：员工在 WorkLog 中提交并经 PM 确认的工时
- **饱和度**：实际工时 / weeklyCapacity
- **偏差率**：(实际 − 预估) / 预估
- **虚拟产值**：仅内部绩效参考，不等于真实合同收入
